# 逻辑谜题生成器 — 开发交接文档

> 目标:构建一个 **纯算法**(非 LLM)的凶案推理逻辑谜题生成器,机制参考 Murdoku。
> 本文档整理自前期调研与技术讨论,可直接作为 Claude Code 的项目起点 / spec。

---

## 1. 一句话定义

一个约束满足型(CSP)逻辑谜题:在 N×N 网格上放置嫌疑人,满足"每行每列唯一"(拉丁方)+ 空间线索(房间 / 墙 / 相邻 / 方位)。生成器保证每题 **唯一解 + 纯逻辑可解(无需盲猜)+ 难度可控**。逻辑骨架 100% 算法,LLM 最多用于主题皮肤润色。

---

## 2. 参考产品调研:Murdoku 架构

以下为对 `murdoku.com/play` 的实测结论(运行时确认,非推测),作为参考基准:

| 层 | 技术 |
|---|---|
| 前端框架 | React 18(`#root` 上确认 `__reactContainer$` fiber) |
| 构建 | Vite(`index-[hash].js/css`,内容哈希,单入口) |
| 包体 | 主 JS ~95KB / CSS ~3KB — **无游戏引擎**,网格用轻量 DOM/SVG 渲染 |
| 部署 | 静态托管的 PWA(manifest + service worker + apple-mobile-web-app-*) |
| 后端 | **Firebase Realtime Database**(`murdoku-c0640`),无自建 REST API |
| 每日题时间对齐 | 从 Firebase 拉服务器时间(`server_time_offset`),防改本地时间刷题 |
| 状态持久化 | **全部 localStorage**,`mrd_` 前缀 |
| 谜题下发 | 预生成、静态,通过 Firebase catalog 同步;前端验证 |
| i18n | 多语言(观测到 `zh_hans`) |

**关键启示**:这是"前端重、后端轻"的 serverless 架构,成本极低。谜题**离线预生成**,运行时只做渲染 + 校验。我们的生成器应做成**独立的离线 pipeline**,产出静态谜题包(JSON),前端只消费。

### Murdoku 观测到的状态编码(可借鉴)

两套并存的存档格式:

1. 紧凑游程编码字符串,如 `"X2-A3-XE6X2-VD9XC2-BX3-"`
   字母=放置的嫌疑人,数字=重复次数(RLE),`X`=排除标记,`-`=分隔。
2. JSON,如:
   ```json
   { "placed": ["char_4", null, ...], "xMarks": [false, true, ...], "notes": [[], [], ...] }
   ```
   `placed`=每格放的嫌疑人,`xMarks`=排除标记,`notes`=候选笔记。

建议我们统一用 JSON 做内部格式,导出时可选 RLE 压缩。

## 3. 谜题机制(规则模型)

- **网格**:N×N,**支持 6×6(最小)~ 9×9(最大)**。
- **实体(猫)**:N 只,记为 `char_0 … char_{N-1}`,**N = 网格边长**。
- **核心约束模型:排列(permutation / N-Queens)**。每行恰好一只、每列恰好一只 → 全盘只有 N 个占用格,构成一个排列矩阵。房间(格子分组)在此之上叠加,一个房间可含多格、可含 0~多只猫。
- **⚠ 排列模型的几何推论(定线索类型的前提)**:任意两只猫必然不同行、不同列 → 曼哈顿距离恒 ≥ 2。因此**"正交相邻"和"同行 / 同列"这两类"猫-猫"线索在本规则下恒不成立,不予采用**。真正成立的邻近关系是**对角相邻**(行列各差 1)。
- **空间元素**:房间(格子分组)、墙 / 窗等固定特征、对角相邻、相对方位(坐标先后)。
- **初版线索类型(已定,4 类)**:
  1. **同房间 / 不同房间**:A 与 B 在(不在)同一房间。核心,信息量大。
  2. **相对方位**:A 在 B 的左 / 右 / 上 / 下——指坐标先后(`A.col < B.col` 等),**不要求相邻**,排列模型下恒可定义。
  3. **在 / 不在某房间**:A 在(不在)房间 X。直接钉死 / 排除,适合做入口线索与难度调节。
  4. **对角相邻**:A 与 B 斜角相邻(行列各差 1)。排列模型下成立,增加空间趣味。
  - (备选,暂不入初版)不在第 K 行 / 第 K 列;距离约束;边 / 角 / 中心位置。
- **主题皮肤(装饰层,与逻辑解耦)**:实体名、房间名、标题、线索自然语言措辞。模板生成即可,后续可选 LLM 润色。**本项目主题已定为「拆家喵」,详见第 10 节。**

## 4. 核心算法 Pipeline(项目主体)

```
生成完整解 → 枚举候选线索池 → 裁剪到唯一解 → 逻辑可解性校验 → 难度分级 → 拒绝采样
```

### 4.1 生成完整解
随机造一个满足所有硬约束的完整布局。小网格用回溯即可,一秒可出上千。

### 4.2 枚举候选线索池
遍历答案,用模板把已知事实翻译成线索陈述。因为来自已知答案,**天然正确**。

### 4.3 裁剪到唯一解(核心)
两种策略:
- **Constructive**:从空开始不断加线索,直到解唯一。
- **Reductive**(推荐,出题更精简):先给一大堆线索,贪心地逐条删,只要删完仍唯一就继续删。

### 4.4 两个求解器(关键设计 — 分工不同)

| 求解器 | 用途 | 特点 |
|---|---|---|
| **完备求解器**(回溯 / SAT / CSP) | 数解的个数(数到 2 即停),验证唯一性 | 只管穷举,不在乎人能否想通 |
| **逻辑推理求解器** | 保证"纯逻辑可解、无需猜";难度打分 | **只用人类可读的推理规则,禁止盲猜** |

唯一性检查实现:把"排除已知解"编码成阻塞子句,再解一遍看是否存在第二个解。

### 4.5 难度分级(纯算法,最见功夫)
难度 ≠ 能否解出,而是 **需要动用多高级的推理**:
- 逻辑求解器用到的最高规则层级(显式唯一格 → 隐式唯一 → 数对 / 区块排除 → 链式排除 …)。
- 推理链长度、是否需要多步前瞻。
- 若一题只能靠盲猜回溯解出 → 判"不好玩",直接丢弃。

**难度档位(已定,5 档)**——主判据是"逻辑求解器需要的最高推理规则层级",网格大小与线索经济度为辅:

| 档位 | 网格(建议带) | 允许的最高推理规则 | 线索经济度 |
|---|---|---|---|
| 入门 very-easy | 6×6 | 显式唯一格 + 直接钉死 / 排除 | 宽松(给足) |
| 简单 easy | 6×6 ~ 7×7 | + 隐式唯一(某猫在某房间只剩一格) | 较宽松 |
| 中等 medium | 7×7 ~ 8×8 | + 数对 / 区块排除 | 适中 |
| 困难 hard | 8×8 ~ 9×9 | + 链式推理(单链) | 偏紧 |
| 专家 expert | 9×9 | 深链 / 多步前瞻(**仍禁盲猜**) | 最紧(最少线索) |

- 判据以"最高规则层级"为主、网格与线索数为辅;带区间可重叠,由分级器实测求解过程后打标。
- **硬红线**:任一档位都必须"纯逻辑可解、无需盲猜";达不到的候选直接拒绝。

### 4.6 拒绝采样(整体形态)
生成器本质是 rejection sampling:批量生成候选,**只保留 `唯一解 + 纯逻辑可解 + 难度落在目标区间` 的**。

## 5. 建议技术选型

| 模块 | 选型 | 备注 |
|---|---|---|
| 生成器语言 | **Python(已定)** | 求解器生态成熟;离线 pipeline 与前端解耦,不需要同构 |
| 约束求解 | 自写回溯(网格小,几十行足够)/ OR-Tools / python-constraint | 初版自写即可,一秒上千题 |
| 唯一性检查 | 阻塞子句 + 重解 / SAT model counter | |
| 前端 | React 18 + Vite | 对齐 Murdoku,轻量 |
| 网格渲染 | DOM/SVG,**不引入游戏引擎** | 逻辑游戏不需要 Canvas/Phaser |
| 状态持久化 | localStorage(`prefix_` 命名) | 参考 Murdoku 的 mrd_ 前缀模式 |
| 谜题下发 | 离线预生成 JSON 静态包 | 前端只消费 + 校验 |
| 后端(可选) | Firebase / 或纯静态 | MVP 可完全无后端 |

## 6. 建议项目结构

```
puzzle-gen/                # 离线生成器(算法主体)
  ├─ model/                # 网格、约束、线索类型定义
  ├─ generator/            # 完整解生成
  ├─ clues/                # 线索枚举 + 模板
  ├─ solver/
  │    ├─ complete.py      # 完备求解器(唯一性)
  │    └─ logical.py       # 逻辑推理求解器(难度 + 可解性)
  ├─ difficulty/           # 难度分级
  ├─ pipeline.py           # 拒绝采样主流程
  └─ export/               # 输出静态 JSON 谜题包
web/                       # React + Vite 前端
  ├─ src/grid/             # 网格渲染(DOM/SVG)
  ├─ src/state/            # localStorage 存档 + 编码
  ├─ src/solver-check/     # 前端解校验
  └─ public/puzzles/       # 消费生成器产出的 JSON
```

## 7. 建议开发阶段(里程碑)

1. **M1 — 骨架**:网格 + 硬约束模型 + 完整解生成 + 完备求解器(能验证唯一性)。
2. **M2 — 线索**:线索类型定义 + 候选池枚举 + reductive 裁剪到唯一解。
3. **M3 — 逻辑求解器**:实现人类推理规则集,保证纯逻辑可解。
4. **M4 — 难度**:难度分级 + 拒绝采样 pipeline,批量产出分级谜题。
5. **M5 — 前端**:React+Vite 渲染 + localStorage 存档 + 解校验,消费 JSON 包。
6. **M6 — 皮肤 / i18n / PWA**:主题模板、多语言、PWA 打包。

## 10. 主题包:拆家喵(Catdoku) + 随机性设计

> 皮肤与逻辑完全解耦。整套主题就是一个可插拔的 **theme pack(数据)**;换主题 = 换这份数据,求解器 / 生成器 / 难度分级一行不动。

### 10.1 叙事框架
回家发现满屋狼藉,N 只猫各自躲在一个房间。玩家根据破坏现场 + 线索,推理出每只猫此刻藏在哪。全部归位 = 破案。

- **whodunit → which-cat-did-it**:保留凶案的推理钩子。
- **揭晓时刻**:解出后复原完整"案发现场"——每只猫 + 它闯的祸,逐只"认罪",对应凶案的凶手揭晓。
- 候选产品名:拆家喵 / 喵拆现场 / 爪印疑云 / Catdoku / Purrdoku。

### 10.2 机制映射(纯皮肤)
| 抽象机制 | 拆家喵皮肤 |
|---|---|
| 嫌疑人 `char_N` | 猫(用品种,视觉天然可区分) |
| 房间(格子分组) | 家里的房间 |
| 相邻 / 方位 / 同房间 / 排除 | 猫口吻的破坏叙事,**类型不变** |
| 案件标题 | 随机生成的萌系标题 |

### 10.3 Theme pack 数据(pool,供随机采样)
```json
{
  "cats":  ["橘猫","布偶","黑猫","三花","暹罗","银渐层","奶牛猫","无毛猫","狸花"],
  "rooms": ["客厅","厨房","卧室","书房","阳台","玄关","浴室","储物间","儿童房"],
  "mess":  ["花瓶","绿植","毛线球","键盘","咖啡杯","卷纸","鱼缸","窗帘","纸箱","充电线"],
  "quirks":["爱打翻东西","爱躲纸箱","爱咬线","爱扒拉小物","爱蹬翻水杯"]
}
```
- `cats`:9 种覆盖到 9×9;每题按网格大小随机抽 N 只。
- `mess`(破坏物):本主题**专属彩蛋**,双重作用——(a) 让线索更萌更自然;(b) 可作网格视觉图标,让空间线索更好读。
- `quirks`:可选,给每只猫随机挂个癖好标签,偏置它关联的破坏物。**纯装饰,与逻辑解耦。**

### 10.4 线索措辞模板(按类型,猫口吻)
措辞映射到第 3 节定稿的 4 类线索,每类配一个措辞池,生成时随机选一条填充。示例:
- **同房间 / 不同房间**:`"{A} 和 {B} 一起窝在同一间搞事"` / `"打翻{物}时,{A} 和 {B} 都在场"` / `"{A} 和 {B} 各拆各的,压根不在同一间"`
- **相对方位**:`"{A} 待的屋子比 {B} 更靠{方位}"` / `"从 {B} 那间往{方位}方向,才轮到 {A}"`(不要求相邻,指坐标先后)
- **在 / 不在某房间**:`"{A} 就赖在{房间}不出来"` / `"{房间}的{物}还好好的,{A} 肯定没来过"`
- **对角相邻**:`"{A} 和 {B} 在斜对角的两间,爪子都快够着了"`

> 注意:措辞不得暗示"正邻 / 同排 / 同列"等在排列模型下不成立的关系(见第 3 节几何推论)。

### 10.5 标题模板(随机)
`"{房间}惨案"` / `"谁动了我的{物}"` / `"{猫}的不在场证明"` / `"周一早晨的{房间}"` / `"{物}失踪之谜"` — 每题随机采样,天然出新。

### 10.6 随机性:两层 + 注入点
- **A. 皮肤随机(procedural skin)**:每题随机采样 cats / rooms / mess / 措辞模板 / 标题模板 / quirks 组合。每题都是新包装。这层 LLM 可选做措辞润色,模板+池子已够。
- **B. 结构随机(puzzle variety)**,生成阶段注入:网格尺寸(难度档内随机)、房间划分与形状、墙的位置、线索类型配比、线索数量。
- **关键**:pipeline 本就是"随机提案 → 严格过滤(拒绝采样)"。增加随机 = 放宽**提案端**,验证端(唯一解 + 纯逻辑可解 + 难度)永远严格。
- **seed 系统**:所有随机由种子驱动,得到"可复现的随机"——每日题 / 分享同一题 / debug 全靠它。

### 10.7 对项目结构的影响
- theme pack 放独立目录(如 `themes/catdoku/`),含 pool 数据 + 措辞模板 + 图标资源。
- 生成器输出的谜题 JSON 只存**逻辑层**(网格 / 房间 / 线索的抽象结构 + 引用的实体 id),皮肤在前端渲染时套用 theme pack。这样一份谜题可换任意皮肤,也方便日后加第二主题。
